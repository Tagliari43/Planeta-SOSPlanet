# 🤝 Diretrizes Sagradas de Colaboração (CONTRIBUTING)

Irmão Construtor, bem-vindo à forja! Ficamos imensamente felizes de ver você unindo suas mãos para a construção do **Santuário SOSPlanet**. Nossa força reside na harmonia das colaborações.

Estas são as diretrizes para participar abertamente da nossa comunidade.

## 🧭 Bússola Rápida

1. Revise nosso \`README.md\` para entender arquitetura e missão.
2. Certifique-se de não sobrepor a ferramenta de um irmão. Sempre verifique as *Issues* Abertas ou as Pranchas do projeto para checar no quê a comunidade está focando.
3. Esteja alinhado com o Código da Nação (nosso Código de Conduta). Respeito, empatia e construção mútua.

## 🛠 Fluxo de Contribuição (O Caminho do Código)

### 1. Criar Issues (Reportando ou Solicitando)
Se você encontrou um bug ou tem uma visão radiante sobre como a SOSPlanet deve avançar:
- Abra uma *Issue* no Github.
- Use etiquetas (labels) que descrevam apropriadamente: `bug`, `enhancement`, `documentation`, `UI/UX`.
- Explique de maneira simples e didática o problema (para garantir que humanos e oráculos artificiais entendam perfeitamente).

### 2. Preparando a Bifurcação (Fork & Branch)
- Faça um **fork** do repositório no Github.
- Clone o fork para o seu espaço de trabalho local.
- Crie um ramo (branch) para trabalhar em sua feature (o nome deve ser direto):
  \`git checkout -b feat/painel-oraculo\` ou \`git checkout -b fix/bug-telemetria\`

### 3. Escrevendo o Código
- Todos somos artesãos. Mantenha os métodos curtos, descritivos e padronizados.
- Se o projeto usar React/TypeScript (como no nosso Santuário Frontend), atenha-se aos tipos explícitos para o seu novo código.
- Assegure-se de que se o novo portal exibir informações críticas ao mundo, não esqueça de projetá-lo de maneira **acessível e mobile-first**.

### 4. Oferecendo à Nação (O Pull Request)
- Se a sua implementação for significativa: acrescente testes para atestar estabilidade.
- Assim que empurrar o branch para o repositório remoto, crie o PR (*Pull Request*).
- O seu PR precisa descrever claramente as intenções construtivas:
  - O que modifica?
  - Quais *issues* referenciadas?
  - Prints de Tela (caso inclua alterações Visuais/UI).

## ✨ Regras do Código Limpo
Nossa arquitetura depende da comunicação amigável, clara e concisa. Qualquer código, variável imperfeita, ou estado complexo, deve vir sempre com comentários e documentações esclarecedoras.
